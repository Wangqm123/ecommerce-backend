# 算法2 — RFM 用户分群 对接文档

> 本文档描述算法2（RFM 用户价值分层）与后端数据库的接口约定、数据流转、部署方式及输出指标。

---

## 一、算法概述

算法2 基于 **RFM 模型** 对用户消费数据进行分析，输出以下结果：

| 输出项 | 说明 |
|--------|------|
| 用户价值分层 | 8 类 RFM 分层标签（重要价值客户、重要发展客户…） |
| 购买力评级 | 5 档（高/中高/中/中低/低购买力） |
| 复购率 | 全体用户中消费 ≥2 次的比例 |
| 留存率 | 两期均活跃用户 / 上期活跃用户的比例 |

---

## 二、任务生命周期

```
┌─────────────┐     ┌───────────────┐     ┌──────────────┐     ┌──────────────┐
│  后端/前端   │     │  rfm_compute  │     │   算法2 服务   │     │  rfm_results │
│  创建任务    │     │   _tasks 表   │     │   (轮询)      │     │   结果表     │
└──────┬──────┘     └──────┬────────┘     └──────┬───────┘     └──────┬───────┘
       │                   │                     │                    │
       │ INSERT status     │                     │                    │
       │ = 'pending' ────► │                     │                    │
       │                   │                     │                    │
       │                   │  SELECT pending ◄───│ 轮询(每5秒)        │
       │                   │                     │                    │
       │                   │  UPDATE running ◄───│ 认领任务           │
       │                   │                     │                    │
       │                   │                     │ 查询订单数据       │
       │                   │                     │ (orders / v_rfm)   │
       │                   │                     │                    │
       │                   │                     │ RFM 评分+分层      │
       │                   │                     │ 购买力/复购/留存   │
       │                   │                     │                    │
       │                   │                     │ INSERT 结果 ──────►│
       │                   │                     │                    │
       │                   │  UPDATE completed ◄─│ 标记完成           │
       │                   │                     │                    │
       │  GET /api/rfm/    │                     │                    │
       │  result ◄─────────│◄────────────────────│                    │
       │                   │                     │                    │
```

### 状态流转

```
pending  →  running  →  completed
                    ↘  failed
```

| 状态 | 含义 | 触发方 |
|------|------|--------|
| `pending` | 等待计算 | 后端 POST /api/rfm/trigger |
| `running` | 计算中 | 算法2 服务认领后 |
| `completed` | 计算完成 | 算法2 写入结果后 |
| `failed` | 计算异常 | 算法2 捕获异常后 |

## 三、涉及的数据库表

### 3.1 读取：订单源数据

算法2 从订单表读取原始数据（优先使用视图 `v_rfm_input`，回退查询 `orders` 表）。

```sql
-- 算法实际执行的查询（当期数据）
SELECT user_id, order_id, order_date, total_amount
FROM orders
WHERE order_status = 'completed'
  AND order_date BETWEEN '<startDate>' AND '<endDate>';

-- 留存率对比查询（上期数据，等长窗口自动推算）
-- 如当期 2025-01-01 ~ 2025-12-31，则上期为 2023-12-31 ~ 2024-12-30
SELECT user_id, order_id, order_date, total_amount
FROM orders
WHERE order_status = 'completed'
  AND order_date BETWEEN '<prevStart>' AND '<prevEnd>';
```

> **依赖约定**：`orders` 表需包含 `user_id`, `order_id`, `order_date`(DATE/DATETIME), `total_amount`, `order_status` 字段。

### 3.2 读取 + 写入：任务表 `rfm_compute_tasks`

| 操作 | SQL | 说明 |
|------|-----|------|
| 轮询 | `SELECT * FROM rfm_compute_tasks WHERE status='pending' ORDER BY created_at ASC LIMIT 1` | 取最早 pending 任务 |
| 认领 | `UPDATE rfm_compute_tasks SET status='running', started_at=NOW() WHERE batch_uuid=?` | 标记执行中 |
| 完成 | `UPDATE rfm_compute_tasks SET status='completed', user_count=?, completed_at=NOW() WHERE batch_uuid=?` | 写入用户数 |
| 失败 | `UPDATE rfm_compute_tasks SET status='failed', completed_at=NOW() WHERE batch_uuid=?` | 异常标记 |

**任务参数**（`params` 字段，JSON 格式）：

```json
{
  "startDate": "2025-01-01",
  "endDate": "2025-12-31",
  "recencyBaseDate": "2025-12-31",
  "scoringMethod": "quantile"
}
```

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| startDate | string | 否 | "2025-01-01" | 分析数据起始日期 |
| endDate | string | 否 | "2025-12-31" | 分析数据截止日期 |
| recencyBaseDate | string | 否 | endDate 的值 | R 值参照日期（距今基准） |
| scoringMethod | string | 否 | "quantile" | 评分方法：`quantile`(五分位) / `fixed`(固定阈值) |

> **注意**：算法2 在 `/api/rfm/trigger` 创建任务后即开始轮询执行；前端需通过 `/api/rfm/status/:batchUuid` 轮询到 `completed` 后再调 `/api/rfm/result` 获取结果。

### 3.3 写入：结果表 `rfm_results`

每次计算前会先 `DELETE FROM rfm_results WHERE compute_batch = ?` 清除同批次旧数据，确保幂等。

```sql
INSERT INTO rfm_results
  (user_id, recency, frequency, monetary,
   r_score, f_score, m_score,
   rfm_segment, rfm_group, compute_batch)
VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| user_id | BIGINT | 用户ID |
| recency | INT | 最近消费距参照日天数 |
| frequency | INT | 消费次数（去重订单数） |
| monetary | DECIMAL(14,2) | 消费总金额 |
| r_score | INT | R 评分 1-5（5=最近消费） |
| f_score | INT | F 评分 1-5（5=消费最频繁） |
| m_score | INT | M 评分 1-5（5=消费金额最高） |
| rfm_segment | VARCHAR(30) | 分层标签（如"重要价值客户"） |
| rfm_group | VARCHAR(10) | RFM 三位组合（如"555"） |
| compute_batch | VARCHAR(36) | 批次UUID |

## 四、算法核心逻辑

### 4.1 RFM 原始值

| 指标 | 计算方式 | 含义 |
|------|----------|------|
| R (Recency) | `(recencyBaseDate - 用户最近订单日期).days` | 距今天数，越小越好 |
| F (Frequency) | `用户去重订单数` | 消费频次，越高越好 |
| M (Monetary) | `用户消费总金额 sum(total_amount)` | 消费金额，越高越好 |

### 4.2 评分方法

**方法一：五分位法 (quantile)** — 默认

将用户按 R / F / M 值分别排序，等频切分为 5 个桶：

- R：最近 20% → 5 分，次近 20% → 4 分，…最远 20% → 1 分
- F：最高 20% → 5 分，…最低 20% → 1 分
- M：最高 20% → 5 分，…最低 20% → 1 分

**方法二：固定阈值法 (fixed)**

| 维度 | 5分 | 4分 | 3分 | 2分 | 1分 |
|------|-----|-----|-----|-----|-----|
| R (天) | ≤30 | ≤60 | ≤90 | ≤180 | >180 |
| F (次) | ≥20 | ≥10 | ≥5 | ≥2 | 1 |
| M (元) | ≥50000 | ≥20000 | ≥10000 | ≥5000 | <5000 |

> 固定阈值可在 `config.py` 的 `FIXED_THRESHOLDS` 中调整。

### 4.3 用户分层（8 类）

按 R / F / M 评分是否 ≥3 分为"高""低"两档，组合为 8 类：

| R | F | M | 分层标签 |
|---|---|---|----------|
| 高 | 高 | 高 | 重要价值客户 |
| 高 | 低 | 高 | 重要发展客户 |
| 低 | 高 | 高 | 重要保持客户 |
| 低 | 低 | 高 | 重要挽留客户 |
| 高 | 高 | 低 | 一般价值客户 |
| 高 | 低 | 低 | 一般发展客户 |
| 低 | 高 | 低 | 一般保持客户 |
| 低 | 低 | 低 | 一般挽留客户 |

### 4.4 购买力评级

基于 M 评分直接映射：

| M 评分 | 购买力等级 |
|--------|-----------|
| 5 | 高购买力 |
| 4 | 中高购买力 |
| 3 | 中购买力 |
| 2 | 中低购买力 |
| 1 | 低购买力 |

### 4.5 复购率

```
复购率 = (frequency ≥ 2 的用户数) / 总用户数 × 100%
```

### 4.6 留存率

算法自动推算上期窗口（与当期等长且紧邻），对比两期均活跃的用户：

```
留存率 = (当期 ∩ 上期 用户数) / 上期用户数 × 100%
```

> 示例：当期 2025-01-01 ~ 2025-12-31（365天），上期为 2023-12-31 ~ 2024-12-30。
> 留存率 = 两期均有订单的用户数 / 上期有订单的用户数。

## 五、部署运行

### 5.1 环境要求

- Python 3.9+
- MySQL 5.7+ / 8.0（`ecommerce_analysis` 数据库已建表）

### 5.2 安装依赖

```bash
cd algorithm2_rfm
pip install -r requirements.txt
```

### 5.3 配置数据库连接

方式一：直接修改 `config.py`：

```python
DB_CONFIG = {
    "host": "localhost",
    "port": 3306,
    "user": "root",
    "password": "your_password",
    "database": "ecommerce_analysis",
    "charset": "utf8mb4",
}
```

方式二：环境变量（推荐生产环境）：

```bash
export DB_HOST=localhost
export DB_PORT=3306
export DB_USER=root
export DB_PASSWORD=your_password
export DB_DATABASE=ecommerce_analysis
export POLL_INTERVAL=5          # 轮询间隔(秒)
export LOG_LEVEL=INFO
```

### 5.4 启动方式

```bash
# 方式一：前台持续运行（生产环境推荐，配合 systemd/supervisor 守护）
python main.py

# 方式二：单次执行一个 pending 任务后退出（调试用）
python main.py --once

# 方式三：指定处理某个任务（重试/补跑用）
python main.py --batch-uuid a1b2c3d4-e5f6-...
```

### 5.5 守护进程示例（systemd）

```ini
# /etc/systemd/system/rfm-algorithm2.service
[Unit]
Description=Algorithm 2 - RFM User Segmentation
After=network.target mysql.service

[Service]
Type=simple
User=algo
WorkingDirectory=/opt/algorithms/algorithm2_rfm
Environment="DB_HOST=localhost"
Environment="DB_PASSWORD=xxx"
ExecStart=/usr/bin/python3 main.py
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
```

## 七、与后端接口的对应关系

| 后端 API | 算法2 操作 | 说明 |
|----------|-----------|------|
| `POST /api/rfm/trigger` | — | 后端插入 pending 任务，算法2 自动发现 |
| `GET /api/rfm/status/:batchUuid` | 更新 `rfm_compute_tasks.status` | 后端轮询状态，算法2 写入 running/completed/failed |
| `GET /api/rfm/result` | 写入 `rfm_results` | 算法2 计算结果后存入，后端读取返回前端 |

算法2 **不直接提供 HTTP API**，而是通过数据库表与后端解耦交互。后端无需知道算法2 的运行地址，只需读写约定的表即可。

## 八、异常处理

| 场景 | 处理方式 |
|------|----------|
| 数据库连接失败 | 日志记录错误，按 `POLL_INTERVAL` 间隔重试 |
| 任务执行异常 | `try/except` 捕获，标记 `status=failed`，打印完整堆栈 |
| 当期订单数据为空 | 标记 `completed`，`user_count=0`（不视为失败） |
| 上期数据查询失败 | 仅跳过留存率计算，RFM 其余指标正常产出 |
| 同一批次重复执行 | 先 `DELETE` 同批次旧结果再 `INSERT`，确保幂等 |
| 多个算法实例同时运行 | 谁先 `UPDATE status=running` 谁认领，另一实例跳过 |

## 九、扩展建议

### 9.1 购买力字段持久化（建议后端添加）

当前 `rfm_results` 表不含购买力字段。若前端需要展示购买力等级，后端可：

1. **方案一（推荐）**：根据 `m_score` 实时映射（规则见 §4.4），无需额外存储。
2. **方案二**：在 `rfm_results` 表中增加 `purchasing_power VARCHAR(20)` 列，算法2 同步写入。

### 9.2 汇总指标获取

复购率、留存率等汇总指标当前仅输出到日志。若前端需要在页面展示，建议后端：

1. 读取 `rfm_results` 表后自行计算（复购率 = `COUNT(*) FILTER(WHERE frequency>=2) / COUNT(*)`）
2. 或增加汇总结果表 `rfm_summary`，算法2 同步写入汇总行。

### 9.3 数据库 Schema 兼容性

API 文档（`接口对接文档.md`）中的数据库为 `ecommerce_analysis`，使用 `orders` 宽表；需求文档（`1.docx`）中的数据库为 `ecommerce_data`，使用 `order_master` + `order_item` 拆分表。

算法2 当前按 API 文档的 `orders` 表实现。若实际部署使用 `order_master` + `order_item` 表，需调整数据查询 SQL（或由后端创建 `v_rfm_input` 视图做适配）。
